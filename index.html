Purpose

The purpose of this project is to develop a privacy-preserving healthcare application that utilises the Private-Set Intersection(PSI) protocol. Specifically, the project seeks to create a system which allows an individual to determine the risk of contracting a particular disease, without the need to reveal their sensitive genetic information to healthcare providers. By implementing this PSI protocol, the system essentially ensures the privacy of both the individual client and the healthcare provider whilst allowing for the disease risk assessment to be carried out based on the intersection of the individual's genetic data and the hospital's set of disease-causing genes.

Introduction: What is Private Set Intersection (PSI)?

Private Set Intersection (PSI) is a secure two-party computation (2PC) problem that allows for two parties to compute the intersection between both sets of data, while the elements that are not in the intersection remain private to the other party. There are many different approaches to this problem, however, for this project, we will be focusing on the Diffie-Hellman based PSI approach.
How can you find an intersection of two private data sets without revealing the contents of each data set?
Preprocessing: We will first generate a cryptographic representation of the private sets through methods such as encryption or hashing. These will be used to hide the actual values in the sets for both parties. Comparison: The cryptographic representation of the 2 sets is compared to determine the intersection. Comparison must be performed in a way that the actual elements in the sets remain confidential. Output: Result of the PSI protocol is the intersected elements. Both parties would know the identity of the intersected elements as they know the original element of the set.

What are the benefits of using PSI?

PSI allows privacy of the information to be preserved, while allowing for a meaningful exchange of information between two parties. No additional information aside from the intersected elements is disclosed,
ensuring both parties‚Äô data remain confidential. This can be useful in multiple ways, and this is explored in the two examples below:

Voting - PSI allows the user to check if a voter is accidentally registered in 2 different states by checking if any registered voters intersect in the list of voters between the two states.
Doing this through the use of PSI maintains the privacy and identity of the other voters to the other state.

Healthcare - PSI allows for the user to compare his/her genes against the hospital‚Äôs database to compute the risk of a particular disease without revealing her genes. On the other hand,
the hospital also does not wish to reveal its discoveries to the user as the hospital has spent time and money researching the genes. As this data is private to both the user and the hospital,
PSI allows for the user to only discover the risk of the disease without revealing additional information.

What are some challenges of implementing PSI in our application?

1. Asymmetric Gene Set Size

The specialised hospital is expected to have a significantly larger set of gene set data as compared to the patient. This would lead to an asymmetric gene set size between the two parties, which may significantly increase the computation time. Certain protocols need to be implemented to ensure the computation process remains efficient. To address this issue, cuckoo hashing can be used to ensure a constant time lookup of the data, ensuring that the application remains efficient.

2. Ensuring Privacy of Gene Sets

A collection of individual gene data is known and stored in a database. If hashing alone is used in the process of PSI, malicious attackers may use dictionary attacks to compromise the privacy of the patient or the hospital. To address this issue, homomorphic encryption can be used to ensure the privacy of both parties. Homomorphic encryption allows for computation of the data while it remains encrypted, allowing both parties to determine the intersected sets without revealing additional information.

What is homomorphic encryption and how can it be used in PSI?

Homomorphic encryption allows mathematical operations on data to be carried out on a ciphertext. There are a few types of homomorphic encryption:

Partially Homomorphic encryption - Only a defined operation can be performed infinite times on the ciphertext. Relatively easy to design.

Somewhat Homomorphic encryption - Limited number of addition or multiplication operations are allowed. More difficult to design a system that supports a set number of operations than one operation infinite times.

Fully Homomorphic encryption - Infinite number of additions or multiplication for ciphertext is enabled. Programs for any functionality can run on encrypted inputs to produce an encrypted output.

What are the problems implementing homomorphic encryption?

Efficiency: Homomorphic encryption is computationally expensive. Performing operations on encrypted data takes significantly more time than on plaintext data.
Resource Intensive: It requires substantial memory and computational resources, especially for fully homomorphic encryption (FHE) schemes.

What is Secure Multi-Party Computation (SMPC)?

Secure Multi-Party Computation deals with the protection of input data of 2 parties communicating with each other. One such method is known as Shamir‚Äôs Secret Sharing. Two (or more) parties secretly distribute a secret (such as an encryption key), with each party obtaining 1 part of the secret share. The secret will remain concealed unless a minimum number of parties (the threshold) collaborate to reconstruct the secret.

What is Cuckoo Hashing?

Cuckoo Hashing is a form of open addressing where each non-empty cell in a hash table contains a key or a key-value pair. It uses 2 hash functions to determine the location for each key and requires a table of size 1.5 times the number of keys. The presence of a key can be found by examining the corresponding cell in the table. To illustrate:

1. We have a hash table of 2 arrays, A and B.
2. When inserting a key, we compute its hash, using both hash functions.
3. If the cell in array A corresponding to the first hash function is empty ‚Üí we place the key there.
4. If the cell in array A corresponding to the first hash function is full ‚Üí we evict the existing key from array A and move it to array B.
5. We will then insert the new key into the vacated cell in array A.
6. If the cell in array B corresponding to the second hash function is also occupied, we repeat the process potentially evicting another key.
7. The cycle continues until we find an empty cell or reach a max number of iterations.
